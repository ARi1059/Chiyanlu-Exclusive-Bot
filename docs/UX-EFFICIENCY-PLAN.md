# UX-EFFICIENCY-PLAN

> 本文档为 **操作效率优化计划**，不是已实现功能列表。
> 文档生成于 plan mode，作者**未**修改任何业务代码、脚本、迁移、测试或已有文档（包括 `docs/ROADMAP-PLAN.md`）。
> **核心目标：减少操作次数，而不是盲目减少按钮数量。**

---

## 0. 文档声明

1. **本文档是 UX 操作效率优化计划**，描述「老师 / 普通用户 / 管理员 / 超级管理员」四类角色在 Bot 内的操作路径如何变得更短、更稳、更不容易出错。
2. **本文档不是已实现功能**。所有条目都需要后续在独立 PR 中评估、设计、实现与验证。
3. **不代表所有优化必须立即开发**。文档提供「优化原则 + Sprint 路线 + 标准模板」，具体哪一项先做由生产观察、运营反馈共同决定。
4. **每一项必须单独评估、单独 PR、单独 CI 验证**。禁止把多个角色、多个 Sprint 的 UX 调整打包到一个 PR。
5. **核心目标重申**：减少操作次数，而不是盲目减少按钮数量。一个被合并到首页的按钮如果用户找不到，那是反向优化。
6. 本文档与 `docs/ROADMAP-PLAN.md` 互补：
   - `ROADMAP-PLAN.md` 关注**功能迭代**（Sprint 1–7：看板对账、报销可视化、积分配置等）。
   - 本文档关注**操作路径**（点击数、入口位置、返回路径、空状态、badge）。
   - 二者**不重复、可并行**，但同一时间窗口内仍要遵循小步提交。
7. **本文档生成于 plan mode**：撰写过程中没有修改 `bot/`、`scripts/`、`update.sh`、`tests/`、`README.md` 或 `docs/` 下其它文件，没有新增数据库迁移，没有执行任何功能实现。

## 1. 优化原则

> 任何 UX 优化都必须同时满足以下七条原则。违反任何一条的 PR，应在 review 阶段被退回。

### 1.1 高频优先

优先优化**每天都有人用**的路径。粗略优先级：

- **极高频**：老师签到、用户找老师入口、用户进入收藏 / 最近看过。
- **高频**：用户搜索、用户提交评价、管理员审核（评价 / 报销 / 老师资料）。
- **中频**：超管查看运营状态、用户查看积分 / 报销 / 抽奖记录。
- **低频**：超管修改规则、超管查看审计日志。

低频路径可以容忍多 1-2 次点击，**绝不能反过来**——不能为了优化低频路径而把高频路径搞复杂。

### 1.2 不破坏旧入口

- 旧 callback / 旧 inline button **必须继续可用**。Telegram 客户端里的按钮是历史快照，用户随时可能点到几个月前发出的消息。
- 修改菜单时优先**新增 callback**，旧 callback 至少保留一个 Sprint。
- 文案修改可以自由进行（只要不改 `callback_data`）。
- 删除旧 callback 必须先经过明确的"双跑观察期"。

### 1.3 不降低权限

- 快捷入口、聚合页、badge 跳转——任何一种 UX 优化，都**不能**让普通管理员看到 super-only 功能。
- 任何一种 UX 优化，都**不能**让普通用户触发管理员级动作。
- 任何一种 UX 优化，都**不能**让老师看到用户隐私或操作权益数据。
- 每个 PR 必须显式声明：谁能看到入口？谁能触发动作？谁能查看结果？

### 1.4 不自动执行高风险操作

以下动作**永远**不允许在 UX 优化中"自动化"：

- 报销通过 / 驳回 / 取消
- 积分人工调整（加 / 扣）
- 抽奖开奖 / 取消
- 删除 / 清空数据
- 修改规则配置（报销月度池、积分规则、抽奖参数）

UX 优化可以让这些动作**更容易找到**，但不能让它们"少一步就触发"。

### 1.5 减少点击不等于减少确认

对高风险操作：

- **可以减少**：找入口的点击、跳转链路的层级、二级菜单到详情页的跳数。
- **不能减少**：触发前的二次确认、`admin_audit_logs` 的写入、POLICY 文档的同步更新。

如果你为了"减少一次点击"而把"确认弹窗"砍掉了，那是事故而不是优化。

### 1.6 先快捷入口，后流程改造

UX 优化按收益 / 风险三段式：

1. **第一段（推荐先做）**：新增快捷按钮、默认值、返回上一级按钮、刷新按钮、空状态引导。
2. **第二段**：调整返回路径、合并菜单分组、统一文案命名。
3. **第三段（最后做）**：重写流程、删除旧入口、引入新交互范式。

不要倒着做。倒着做 = 先重写再发现旧入口有人用 = 回滚成本极高。

### 1.7 可回滚

- 每次 UX 优化都必须能通过 `update.sh` 的回滚机制退回。
- 任何涉及 schema 变更的 UX 优化（极少见，例如新增 badge 表）必须走 `MIGRATIONS.append(Migration(...))`，禁止直接 `init_db()` 内追加。
- 任何涉及 config 表变更的 UX 优化必须有"旧值兜底"逻辑——读不到新 key 时回退到旧行为。

## 2. 角色一：普通用户操作效率优化

### 2.1 高频任务

- 找老师（热门 / 今日可约 / 条件筛选 / 搜索）
- 查看「今日可约」
- 查看「最近看过」
- 查看「我的收藏」
- 重复使用历史搜索条件
- 提交评价
- 查看积分 / 报销 / 抽奖记录

### 2.2 当前可能的操作成本

> 站在用户视角说，没有具体定位代码位置（属于实施阶段的事）。

- **找老师入口可能分散**：用户进入首页后需要判断「我要点哪个按钮才能开始找」。
- **搜索历史 / 最近看过 / 收藏之间跳转链路偏长**：从详情页返回时，可能跳回首页而不是来源列表。
- **提交评价后缺少"下一步"引导**：用户不知道评价能不能加分、报销资格是否变化、是否可以参与抽奖。
- **积分 / 报销 / 抽奖入口分散**：散落在首页多个一级按钮中，用户找不到自己的"账户中心"。

### 2.3 优化方向

> 每一项都是**建议**，不是 hard requirement，具体方案由对应 PR 设计。

#### A. 🔎 找老师 聚合入口

把以下入口**收纳**进「🔎 找老师」一级菜单：

- 热门推荐
- 今日可约
- 条件筛选
- 搜索历史

**约束**：

- 保留原入口至少一个 Sprint（双跑期）。
- 不修改搜索算法、不修改排序、不修改"今日可约"判断逻辑。
- 旧 callback 在原位置仍然有效。

#### B. 老师详情页快捷动作

在老师详情页底部固定一行快捷按钮：

- ⭐ 收藏 / 取消收藏（按当前状态切换文案）
- 📝 写评价
- 👀 最近看过返回
- 🔎 继续搜索
- ⬅️ 返回上一级（按"来源页"动态跳转，不是固定回首页）

**约束**：

- 不修改老师详情页的核心信息呈现。
- "返回上一级"必须按 source 动态跳转，避免把所有用户都甩回首页。

#### C. 用户「下一步动作」引导

在关键完成态后给出**最多 2-3 个下一步**：

- 搜索无结果 → 「条件筛选」/「热门推荐」/「修改搜索条件」
- 收藏成功 → 「查看我的收藏」/「返回详情」
- 评价提交成功 → 「查看积分」/「查看报销资格」/「继续找老师」

**约束**：

- 不让用户感到被推着走，按钮永远是可选项而不是强制跳转。
- 不在"提交评价成功"页直接弹出"立即报销"按钮（视为变相诱导）。

#### D. 我的记录聚合

把以下入口收纳到「📝 我的记录」二级菜单：

- 我的评价
- 我的报销
- 积分流水
- 抽奖记录

**约束**：

- 不合并报销 / 积分 / 抽奖的查询逻辑（仍各自从自己的表查）。
- 不改任何权益数据。

#### E. 空状态优化

对于：

- 最近看过为空
- 我的收藏为空
- 搜索历史为空
- 我的评价为空
- 报销记录为空

**统一给出可点击的下一步**，例如：

- 「去逛热门老师」
- 「去搜索」
- 「了解评价怎么写」

不要只显示"暂无数据"四个字。

### 2.4 不建议做的优化

- 不自动帮用户报名抽奖。
- 不自动提交评价、不自动生成评价文本。
- 不自动收藏用户访问过的老师。
- 不把所有功能塞到首页（首页一级按钮 ≤ 6 个）。
- 不隐藏「找老师」核心入口（即使做了聚合，也要确保用户立刻看到）。
- 不为了"减点击"而把"提交评价"前的确认步骤砍掉。

### 2.5 验收标准

- 用户从首页进入老师详情**平均不超过 2-3 次点击**。
- 用户从详情页收藏老师后，能**1 次点击**回到「我的收藏」页。
- 用户进入「最近看过 / 我的收藏 / 搜索历史」任一空状态时，都有**至少一个**可点击的下一步入口。
- 旧 callback 仍可用（手动点击历史 inline button 不报错）。
- CI 全绿，生产 `healthcheck.sh` 0 ERR。

## 3. 角色二：老师操作效率优化

### 3.1 高频任务

- 今日签到
- 修改个人资料
- 查看自己的展示页（用户视角）
- 查看最近评价 / 反馈
- 联系管理员

### 3.2 当前可能的操作成本

- 签到入口不够突出，老师每天进来需要找。
- 修改资料步骤较多，常改的字段（价格、地区、时间、状态）和不常改的字段混在一起。
- 老师不知道自己今天在用户侧实际呈现什么样。
- 评价反馈查看不够方便，且没有"摘要 / 最近一条"的快速预览。

### 3.3 优化方向

#### A. 签到优先

老师面板**第一按钮固定为**：

```
✅ 今日签到
```

- 今日已签到 → 文案改为「✅ 今日已签到」并不可重复点击或显示签到时间。
- 未签到 → 直接可点。

**约束**：

- 不修改签到逻辑、不修改签到积分规则。
- 即使做了"签到优先"，也要保留任何已有的签到入口。

#### B. 签到后下一步

签到成功后给出最多 3 个下一步：

- 查看今日展示状态
- 修改资料
- 返回老师面板

**约束**：

- 不弹出广告 / 推广信息。
- 不强制跳转。

#### C. 我的资料快捷编辑

把**高频变更字段**做成快捷入口：

- 价格
- 地区
- 服务时间
- 照片 / 简介
- 今日状态（在线 / 忙碌 / 暂停）

**约束**：

- 不允许老师绕过资料审核流程修改受审字段。
- 不修改字段持久化逻辑。
- 不让老师修改自己不该改的字段（积分余额、报销资格等）。

#### D. 老师展示页预览

老师可以**一键查看自己在用户侧的展示效果**（与普通用户看到的呈现一致）。

**约束**：

- 不改用户侧详情页逻辑。
- 不让老师在自己的展示页上看到非展示字段。

#### E. 评价反馈入口

老师可快速查看：

- 最近 N 条评价摘要
- 评价数总数 / 加分总数
- 是否有待处理的差评（如已有此流程）

**约束**：

- 不允许老师修改 / 删除评价。
- 不暴露评价用户的隐私信息。

### 3.4 不建议做的优化

- 不给老师查看用户隐私数据（联系方式、报销记录、积分流水）。
- 不给老师操作报销 / 积分 / 抽奖后台。
- 不把管理员功能下放给老师。
- 不允许老师绕过资料审核。
- 不让老师能直接看到其他老师的运营数据。

### 3.5 验收标准

- 老师从 `/start` 到完成签到**不超过 1-2 次点击**。
- 签到成功后有可见的下一步入口。
- 修改常用资料（价格 / 地区 / 时间 / 状态）入口清晰，**不超过 2 次点击**。
- 老师侧一级菜单**不超过 5 个按钮**：✅ 今日签到 / 👤 我的资料 / 📝 我的评价 / 📢 发布状态 / ❓ 帮助。
- CI 全绿，生产 `healthcheck.sh` 0 ERR。

## 4. 角色三：管理员操作效率优化

### 4.1 高频任务

- 查看运营数据（运营总览 / 报销池 / 抽奖状态）
- 审核评价
- 审核报销
- 查看 queued 报销
- 查看老师今日状态 / 待审老师资料
- 查看抽奖状态
- 处理异常事件

### 4.2 当前可能的操作成本

- 管理员六大分组已完成，但**部分子页面返回**仍直接跳 `menu:main`，而不是回到对应的二级菜单。
- 数据看板存在**命名混淆**：
  - `dashboard:enter`（旧）= 偏用户事件 / 审计 / 历史分析
  - `admin:dashboard`（新）= 运营总览 / 报销池 / 抽奖状态
  - 两个按钮文案接近但含义不同。
- 审核处理入口**缺少 badge**，管理员不知道目前有多少条待办。
- 审核完成后**缺少"下一条"快捷入口**，每条审完都要返回列表再点。

### 4.3 优化方向

#### A. 看板命名优化（与 `ROADMAP-PLAN.md` Sprint 1 同步）

- `dashboard:enter` 按钮文案 → `📈 数据分析`
- `admin:dashboard` 按钮文案 → `📊 运营看板`

**约束**：

- **只改文案，不改 callback_data**。
- 不改两个看板内的任何业务逻辑。

#### B. 子页面返回路径优化（分批做）

将部分子页面的返回按钮从 `menu:main` 调整为对应二级菜单：

- 运营总览 → 返回 `admin:dashboard`
- 报销池状态 → 返回 `admin:dashboard`
- 抽奖状态 → 返回 `admin:dashboard`
- 抽奖管理 → 返回 `admin:operations`
- 积分管理 → 返回 `admin:operations`
- 系统配置子项 → 返回 `admin:settings`

**约束**：

- **必须分批做**，一次不超过 1-2 个二级菜单下的子页。
- 同时保留"返回首页"按钮作为兜底（避免用户在某些情况下找不到出口）。
- 不一次性全改。

#### C. 审核待办 badge

在「✅ 审核处理」入口显示总待办数：

```
✅ 审核处理 (12)
```

可在二级页展示分项：

```
- 老师资料待审 (3)
- 评价待审 (5)
- 报销待审 (4)
```

**约束**：

- 有数量才显示 `(N)`，0 不显示。
- queued 报销是否计入待办数需单独定义（建议**不计入**主 badge，避免误导审核员）。
- badge 查询不能在每次进菜单都全表 count（建议有缓存或限制频率）。

#### D. 审核完成后的下一步

任何审核（通过 / 驳回）完成后，给出最多 3 个下一步：

- 处理下一条（同类型）
- 返回审核列表
- 返回审核处理首页

**约束**：

- "下一条"按钮**不预先 commit**任何动作，仍是普通跳转。
- 不绕过权限校验。

#### E. 运营异常快捷跳转

在运营总览 / 报销池 / 抽奖状态中，对异常项提供跳转：

- 「待审核报销 (N)」 → 报销审核
- 「queued 报销 (N)」 → 报销名单
- 「active 抽奖 (N)」 → 抽奖管理

**约束**：

- 异常 badge 必须严谨，不要把"正常的 queued"标成异常。
- 跳转只是导航，不预设任何操作。

### 4.4 不建议做的优化

- 不自动通过审核（无论评价、报销还是老师资料）。
- 不自动发放报销。
- 不自动处理 queued 报销。
- 不自动开奖。
- 不把超管功能（规则配置、管理管理员、审计日志深度查看）暴露给普通管理员。
- 不为了"减点击"而砍掉"通过 / 驳回"的二次确认。

### 4.5 验收标准

- 管理员从 `/admin` 到待办审核**不超过 2 次点击**。
- 审核完成后能**1 次点击**进入"下一条"。
- 普通管理员视角下 super-only 功能完全不可见。
- 老师 / 普通用户视角下管理员入口完全不可见。
- 旧 callback 仍可用。
- CI 全绿，生产 `healthcheck.sh` 0 ERR。

## 5. 角色四：超级管理员操作效率优化

### 5.1 高频任务

- 查看全局运营状态
- 管理管理员（增 / 删 / 改权限）
- 配置报销 / 积分 / 抽奖规则
- 查看审计日志
- 处理系统异常
- 更新部署后检查（version / migrations / healthcheck）

### 5.2 当前可能的操作成本

- 超管功能已经分组，但**规则配置仍分散**——报销在一处、积分在一处、抽奖在一处，没有统一入口。
- 缺少**统一的规则只读页**，超管想"快速核对当前所有规则"必须翻多个菜单 + POLICY 文档。
- 审计日志与管理员设置之间的衔接弱，看某个管理员"做了什么"需要多步跳转。
- 运维命令（`update.sh` / `healthcheck.sh` / `backup.sh`）只在文档里，Bot 内没有提示。

### 5.3 优化方向

#### A. 超管运营控制台（只读聚合）

提供一个**只读**的"超管运营控制台"，聚合：

- 报销规则（开关 / 月度池 / 最低积分 / 每周限制 / queued 模式 / reset voucher 规则）
- 积分规则（评价通过加分 / 优质评价加分 / 抽奖门票扣分 / 报销最低门槛 / 活动积分规则）
- 抽奖规则（当前活跃活动 / 参与门票 / 中奖规则）
- `schema_migrations` 状态（最新 baseline / 是否有 failed）
- DB health 摘要（DB 体积 / WAL 大小 / 表行数概览）

**约束**：

- **第一版严格只读**。
- 不在控制台上提供任何"修改 / 触发"按钮。
- 不执行任何服务器命令。

#### B. 规则配置入口整合（先只读，后编辑）

后续逐步推进：

1. 报销规则只读 → 编辑（写 `admin_audit_logs`、二次确认、同步 `POLICY-reimbursement.md`）
2. 积分规则只读 → 编辑（优先 config 表，必要时走 `MIGRATIONS`，同步 `POLICY-points.md`）
3. 抽奖规则只读 → 编辑（同步 `POLICY-lottery.md`）

**约束**：

- 每个"只读 → 编辑"过渡都是一个独立 Sprint，**不合并**。
- 编辑动作必须二次确认 + 写 `admin_audit_logs` + 同步 POLICY 文档。

#### C. 审计日志快捷查看

从「🛡 管理员设置」更快进入审计日志：

- 最近 N 条操作日志
- 按管理员筛选
- 按动作类型筛选

**约束**：

- 不修改 `admin_audit_logs` 表结构。
- 不允许超管删除审计日志条目。
- 不允许导出文件（如需统计，复制文本即可）。

#### D. 部署后检查提示

Bot 内（**只读展示**）：

- 当前 git commit 短哈希
- `healthcheck.sh` 最近一次摘要（如有缓存）
- `schema_migrations` 当前 failed 数

**约束**：

- **不在 Bot 内执行** `update.sh` / `healthcheck.sh` / `backup.sh` / `prune.sh`。
- **不在 Bot 内修改** `.env`。
- 数据来源是文件 / 数据库的只读读取，不调用 shell。

### 5.4 不建议做的优化

- 不在 Bot 内直接执行 `update.sh`、`scripts/healthcheck.sh`、`scripts/backup.sh`、`scripts/prune.sh`。
- 不在 Bot 内直接执行 rollback。
- 不让 Bot 修改 `.env`。
- 不自动修复数据库（任何 `point_transactions` / `users.total_points` 差异修正都必须人工二次确认）。
- 不把系统命令暴露给普通管理员。
- 不为了"减点击"而把规则编辑的二次确认砍掉。

### 5.5 验收标准

- 超管从 `/admin` 到核心配置（任一规则只读页）**不超过 2-3 次点击**。
- super-only 入口在普通管理员视角下**完全不可见**。
- 高风险操作（任何配置编辑）保留二次确认。
- 所有规则编辑动作写入 `admin_audit_logs`，且同步更新对应 `POLICY-*.md`。
- CI 全绿，生产 `healthcheck.sh` 0 ERR。

## 6. 跨角色通用 UX 优化

### 6.1 返回路径规则

统一规则：

- **一级页**：返回首页 / 后台首页（`menu:main` 或 `admin:enter`）。
- **二级页**：返回对应一级菜单（不再直接跳首页）。
- **详情页**：返回**来源页**（按 source 动态决定，例如从"我的收藏"进入详情 → 返回我的收藏；从"搜索结果"进入 → 返回搜索结果）。
- **旧按钮**：行为保留至少一个 Sprint。

**禁止**：

- 把所有返回按钮统一改为"回首页"——会让用户在多层菜单中迷路。
- 用户在一个会话内被多次甩回首页 → 视为反向优化。

### 6.2 刷新按钮规范

所有**看板 / 列表页**应有：

```
🔄 刷新
```

约束：

- 刷新不重新发消息，而是 `edit_message_text`（避免消息列表被刷屏）。
- 刷新有最短间隔（防止滥用，例如 ≥ 3 秒）。
- 看板数据查询如果是重计算（如运营总览的聚合），刷新必须有 loading 状态提示。

### 6.3 空状态规范

所有空状态都必须给"下一步"：

- 最近看过为空 → 「去逛热门」/「去搜索」
- 我的收藏为空 → 「去找老师」
- 搜索历史为空 → 「去搜索」
- 我的评价为空 → 「找老师」/「了解评价怎么写」
- 报销记录为空 → 「查看报销规则」
- 审核列表为空（管理员）→ 「返回审核处理」

**禁止**：

- 只显示"暂无数据"四个字然后挂在那。

### 6.4 badge 规范

待办数量统一规则：

- **有数量才显示 `(N)`**，例如 `✅ 审核处理 (3)`。
- **数量为 0 时不显示**，例如 `✅ 审核处理`（不要显示 `(0)`）。
- 不显示具体明细，明细在二级页。
- queued 报销是否计入主 badge 需在实施 PR 内**单独定义**——建议不计入主 badge（避免误导），但在报销审核二级页内显示 queued 子 badge。
- badge 查询不允许全表 count 触发性能问题（必要时加 cache 或限制查询范围）。

### 6.5 文案规范

避免多个入口同名或语义相近：

- `📈 数据分析` ↔ `📊 运营看板`（已规划）
- 避免「我的评价」既出现在用户侧又出现在老师侧时含义不同——必须区分按钮文案。
- 避免「审核」泛指多种动作——明确写出"评价审核 / 报销审核 / 老师资料审核"。
- 同一动作在不同入口的文案应一致（"提交评价" vs "写评价" vs "评价老师" 应统一为一种说法）。

### 6.6 高风险确认规范

以下操作**必须二次确认**：

- 报销通过 / 驳回 / 取消
- 积分人工调整（加 / 扣）
- 抽奖开奖 / 取消
- 删除 / 清空数据（任何表）
- 修改规则配置（报销 / 积分 / 抽奖）
- 删除 / 降级管理员

确认形式：

- 弹出确认按钮，至少显示「确认 / 取消」。
- 高风险动作（删除、清空、规则修改）应显示**操作摘要**和**影响范围**。
- 确认后必须写 `admin_audit_logs`（who / when / before / after / reason）。

## 9. 风险控制

| 风险 | 触发场景 | 防线 |
| --- | --- | --- |
| **旧 inline button 失效** | 修改 callback_data / 删除旧 callback 处理函数 | 文案与 callback 解耦；旧 callback 保留至少一个 Sprint；pytest 覆盖 callback router；上线后手动验证 |
| **普通管理员看到超管功能** | 快捷入口 / 聚合页未做权限校验 | 每个新入口在 PR 内显式声明可见角色；CI 中维持权限测试；多账号手动验证（超管 / 普管 / 老师 / 用户） |
| **用户找不到原入口** | 一次性砍掉旧入口、聚合分组过深 | 聚合先新增、再观察、最后撤旧；旧入口双跑期 ≥ 1 Sprint；保留"返回首页"兜底 |
| **高风险操作被简化过头** | 为了"减点击"砍掉二次确认 | 第 1.5 节硬性区分"减入口"vs"减确认"；高风险动作清单（第 6.6 节）必须二次确认；PR review 强制检查 |
| **返回路径改变导致迷路** | 一次性把所有"返回首页"改成"返回二级菜单" | 分批做（每次 1-2 个二级菜单）；保留"返回首页"按钮作为兜底；上线后观察 24-72 小时再推进下批 |
| **删除功能入口导致误判** | 把"看起来没人点"的入口直接删掉 | 任何"删除入口"都必须经过双跑期 + 数据观察（callback 命中次数）；分阶段下线（先隐藏 → 再删除函数） |
| **菜单分组过深导致点击反而增加** | 为了"分组好看"把每个功能都嵌套到二级 / 三级 | 一级菜单 ≤ 6 个按钮；高频功能保持 ≤ 2 次点击可达；以"减少操作次数"为最终判定（第 0 章） |

每条风险一旦触发，回退顺序：

1. **代码级**：PR revert。
2. **服务级**：`update.sh` 自带的备份回滚机制。
3. **数据级**：`scripts/backup.sh` 产物 + 手动恢复（最后手段，本文档场景下几乎用不到——UX 优化原则上不动权益数据）。

## 10. 当前不做事项

> 明确写出，避免后续讨论时反复绕回。

- **一次性重做三端面板**（用户 / 老师 / 管理员任一端）。必须按 Sprint UX-1 → UX-6 分步推进。
- **自动审核**（评价 / 报销 / 老师资料）。
- **自动报销**（无论是按额度自动放行还是按规则自动通过）。
- **自动开奖** / **自动取消抽奖**。
- **自动扣分 / 退分**。
- **Bot 内执行服务器命令**（`update.sh` / `healthcheck.sh` / `backup.sh` / `prune.sh` / `git pull` / 任何 shell 命令）。
- **修改 `.env`**。
- **直接清空搜索历史**（无二次确认的批量删除）。
- **自动删除 `user_events`**（任何"无人值守"的清理都不允许；清理走 `prune.sh --confirm` 受控流程）。
- **降低超管权限**（任何"为了减点击"而把超管动作的二次确认 / audit log 砍掉的提议）。
- **导出文件**（运营对账、审计日志、积分流水的导出都不在本计划范围；如有必要由独立 PR 评估）。
- **在 Bot 内嵌入实时聊天 / 私信 / 客服系统**。
