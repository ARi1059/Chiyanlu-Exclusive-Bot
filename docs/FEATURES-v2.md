# Chiyanlu-Exclusive-Bot 功能增量需求文档 v2

> 本文档为 v1 (`DESIGN.md`) 的功能增量。v1 不变,本文档定义新增功能、与 v1 的交互规则、以及实现时的关键约束。
>
> 本版本聚焦**用户侧体验**:让普通用户(消费者)从"被动看频道"升级为"私聊互动 + 收藏 + 通知"。同时为老师增加自助管理能力。

---

## 0. 概述

### 0.1 v2 解决的问题

v1 的 bot 是**广播式**结构:14:00 频道发布 + 群组关键词响应。普通用户在系统里没有"账户感",找不到老师只能精准搜索。老师只能签到,改信息要找管理员。

v2 引入用户身份层、收藏关系、自助管理流程。bot 从"广播工具"演进为"互动平台"。

### 0.2 覆盖范围

| 模块 | v1 状态 | v2 变化 |
|---|---|---|
| 用户收藏 | 无 | **新增** |
| 14:00 自动发布 | 频道单向发布 | **保留 + 增量**:同时给收藏者推送私聊聚合通知 |
| 老师签到 | 仅签到 | 不变 |
| 老师信息管理 | 仅管理员可改 | **新增**:老师可自助修改,事后审核 |
| 群组关键词响应 | 精准匹配 | **保留**:逻辑不变,卡片增加"收藏"按钮 |
| 私聊功能 | 仅管理员后台 | **新增**:普通用户私聊体验、组合搜索 |
| `/start` 命令 | 仅管理员有响应 | **新增**:非管理员看到"今日老师 / 我的收藏 / 收藏开课" |

### 0.3 决策路径

本文档的所有功能边界源自一轮设计讨论,关键决策记录在第 6 节。**实现时如有歧义,优先回到第 6 节查决策来源**,而非在代码里临时拍板。

## 1. 需求总览与依赖关系

### 1.1 四大需求 + 一个改造点

| # | 需求 | 优先级 |
|---|---|---|
| F1 | 用户收藏老师 | 高 |
| F2 | 14:00 给收藏者推送聚合通知 | 高 |
| F3 | 老师自助管理资料 + 管理员审核 | 中 |
| F4 | 私聊组合搜索(群组关键词功能保留) | 高 |
| C1 | `/start` 给非管理员的私聊冷启动改造 | 高(F1/F4 的入口前置) |

### 1.2 依赖关系

```
F4 (私聊搜索) ──┐
                ├─→ F1 (收藏) ──→ F2 (开课通知)
C1 (/start)  ──┘

F3 (自助管理 + 审核) ── 独立模块,不依赖 F1/F2/F4
```

**含义**:
- F1 的"收藏入口"挂在搜索结果卡片上 → F4 先行
- C1 是用户对 bot 的第一印象 → 必须和 F4 同步上线,否则用户进 bot 一片空白
- F2 消费 F1 的数据 → F1 完整后才能联调
- F3 完全独立,可以和 F1/F2/F4 并行开发

### 1.3 Telegram 平台约束

实现前需了解的硬约束(影响产品形态):

1. **bot 主动给用户发私聊消息,前提是用户曾经 `/start` 过 bot**
   - 群里点"收藏" → 用户从未私聊过 bot → 无法主动确认 → 需要 deep link 引导
2. **私聊场景里 mention 不强制响铃**
   - 仅视觉高亮,不能绕过用户主动设置的静音
3. **bot 全局发送限速 ~30 msg/sec**
   - 14:00 通知大规模并发时必须分批发送

## 2. 需求详细规格

### 2.1 F1:用户收藏老师

#### 2.1.1 业务目标

让普通用户标记自己感兴趣的老师,作为后续**定向通知**(F2)和**快速访问**(C1)的数据基础。

#### 2.1.2 入口

| 场景 | 是否提供收藏按钮 |
|---|:---:|
| 14:00 频道自动发布消息 | ❌ 不提供 |
| 群组关键词响应卡片(模式 A,单老师卡片) | ✅ 提供 |
| 私聊搜索结果卡片 | ✅ 提供 |
| 私聊"我的收藏"列表的每位老师 | ✅ 提供"取消收藏"按钮 |
| 命令入口(如 `/favorites`) | ❌ 不开放,只通过按钮 |

> **频道不加收藏按钮的理由**:频道作为"广播导流"形态,按钮密度低更易扫读。"收藏"是带主动意图的动作,放在"用户主动搜索"场景更合理。

#### 2.1.3 已收藏 vs 未收藏的状态显示

收藏按钮的状态显示**因场景而异**:

**私聊场景**(私聊搜索结果卡片、"我的收藏"列表):

- 未收藏:`⭐ 收藏`
- 已收藏:`✅ 已收藏(点击取消)`
- 卡片对**单一用户**可见,按钮状态可针对该用户渲染
- 点击行为是**切换**——未收藏点击 = 添加,已收藏点击 = 取消

**群组场景**(群组关键词响应卡片):

- 按钮**始终显示** `⭐ 收藏`,不区分用户
- **原因**:群组卡片对所有群成员可见,无法针对每个个人渲染不同的按钮状态
- 用户当前操作的实际结果通过 **callback alert** 反馈:
  - 未收藏 → 点击 → 写入收藏 → alert 显示 `✅ 已收藏 夏亦菲`
  - 已收藏 → 点击 → 移除收藏 → alert 显示 `已取消收藏 夏亦菲`
- 点击行为仍是**切换**,但状态信息只在 alert 里返回给当前点击用户

> **实现注意**:
> - 私聊场景渲染卡片时需要查询当前用户对该老师的收藏状态(用于按钮文本)
> - 群组场景渲染卡片**不需要**查询收藏状态(按钮文本恒定),只在 callback handler 里查询当前状态做切换
> - 这意味着群组卡片的渲染没有"针对个人"的性能压力,Step 3 实现时也无需为群组做 batch 查询缓存

#### 2.1.4 群组场景的 Deep Link 流(混合方案)

**问题**:群里点"收藏"按钮时,bot 收到的 callback 中包含点击者的 user_id,但**该用户可能从未私聊过 bot**——后续 14:00 推送通知会失败。

**处理流程**:

```
用户在群里点击 [⭐ 收藏 夏亦菲]
        ↓
bot 立即写入 favorites(user_id=点击者, teacher_id=夏亦菲)
        ↓
bot 在群里发一条临时回复(answerCallbackQuery alert 或临时 message):
   "✅ 已收藏夏亦菲"
        ↓
检查该用户是否私聊过 bot(users.last_started_bot)
        ↓
   ├─ 已私聊 → 流程结束
   └─ 未私聊 → 在该 alert 里追加文本:
              "⚠️ 请点击下方按钮激活通知,否则 14:00 收不到开课提醒
              [📩 激活通知]"
              按钮跳转 deep link: t.me/<bot_username>?start=activate
              用户点击 → 跳到 bot 私聊 → 自动 /start →
              users.last_started_bot=1
```

> **关键决策**:无论用户是否激活通知,**收藏数据都立即写入**。这保证用户的收藏意图不流失。激活仅决定"能否收到 14:00 推送"。

#### 2.1.5 取消收藏

- 群组卡片/私聊搜索卡片:已收藏的按钮显示 `✅ 已收藏(点击取消)`,点击即取消
- 私聊"我的收藏"列表:每位老师有"取消收藏"按钮,点击即移除
- 取消收藏不需要二次确认(操作可恢复,且预期会有"反复收藏/取消"的真实使用模式)

#### 2.1.6 边界情况

| 情况 | 处理 |
|---|---|
| 老师被**停用**(`is_active=0`) | 收藏关系**保留**。老师恢复时,收藏继续生效 |
| 老师被**删除** | 收藏关系**级联清理**(数据库外键 ON DELETE CASCADE) |
| 用户点击群里收藏按钮但 bot 写入失败 | 在 alert 里提示"操作失败,请重试" |
| 同一用户重复收藏同一老师 | 数据库唯一索引保证幂等(无副作用) |

#### 2.1.7 数据约束

- 收藏数量:**当前不设上限**(实际中单用户收藏 50+ 的概率低,后续如有滥用迹象再加限制)
- 老师本人**不能查看**自己被多少人收藏(避免老师"刷收藏"的行为)
- 管理员后台**可以**查看收藏总数和热门老师 TOP N(运营数据,不在本文档范围)

### 2.2 F2:14:00 收藏者开课通知

#### 2.2.1 触发时机

**14:00 整点**(与频道自动发布**同一触发点**),对每个收藏者发送一条聚合通知。

#### 2.2.2 聚合规则

对每个**有效收藏者**(已私聊过 bot,即 `users.last_started_bot=1`):

```
当天该用户的"通知集合" = 收藏列表 ∩ 当天已签到老师列表
```

- 集合非空 → 推送一条聚合消息(包含集合中**所有**老师)
- 集合为空 → **静默不推送**(防打扰)

> **聚合而非 fanout 的目的**:每个用户当天最多收到一条 bot 消息。100 个收藏老师同时签到,用户也只收到一条,不刷屏。

#### 2.2.3 消息模板

```
@{用户 mention} 你的收藏老师 2025-05-11 开课共 {N} 位:

[夏亦菲]  [林清雪]
[苏暮晚]
```

**字段说明**:

- `{用户 mention}`:使用 `<a href="tg://user?id={user_id}">{first_name}</a>` 的 HTML 形式,在私聊里产生视觉高亮
  - 注:**这不能绕过用户主动设置的私聊静音**——只是显眼,不响铃
- `2025-05-11`:**当天日期**,格式严格 `YYYY-MM-DD`,使用 `TIMEZONE` 环境变量定义的时区
- `{N}`:今天命中的收藏老师数量
- 按钮:每个签到的收藏老师一个按钮,按钮文本为 `display_name`(或 `button_text`,如已配置),点击跳转到 `button_url`
- 按钮排版:Telegram inline keyboard 自动每行最多 2-3 个按钮(实现时定)

#### 2.2.4 推送限速队列

**工程必做**:

- Telegram 全局限速 ~30 msg/sec,聚合后假设 1000 个用户都需要推送
- **不能并发拉满**——必须用队列分批发送
- 推荐:`asyncio.Queue` + 单消费者循环 + 25 msg/sec 限速(留 5 余量)
- 1000 个用户在 14:00 整点全部推送,耗时约 40 秒——**用户感知是"陆续到达",不是瞬间**

#### 2.2.5 失败处理

| 失败类型 | 处理 |
|---|---|
| 用户屏蔽了 bot(`Forbidden: bot was blocked`) | 标记 `users.last_started_bot=0`,跳过本次推送,下次也不再推 |
| 用户 chat 已删除(`Bad Request: chat not found`) | 同上 |
| 临时网络/服务器错误 | 重试 1 次,仍失败则跳过(不阻塞队列) |
| 推送日志 | 建议简单记录每日推送数、失败数,异常时通知管理员 |

> **当前不做**:重发失败队列、用户级"上次推送时间"追踪。这些是后续功能。

#### 2.2.6 与频道发布的执行关系

14:00 触发时,两件事**并行执行**:

1. 频道自动发布(v1 已有功能)
2. 收藏通知聚合 + 推送(v2 新增)

二者**互不依赖**——频道发布失败不阻断通知推送,反之亦然。但建议在同一个 scheduler job 里编排,共享当日签到数据查询(避免重复扫库)。

### 2.3 F3:老师自助管理 + 管理员审核

#### 2.3.1 业务目标

降低管理员负担(不用为每个老师手动改信息),同时保留管理员的内容质量控制权。

#### 2.3.2 可改字段

| 字段 | 老师可自助改 | 锁定原因 |
|---|:---:|---|
| `user_id` | ❌ | 身份锚定,改了等于换人 |
| `username` | ❌ | 身份锚定 |
| `display_name`(艺名) | ✅ | |
| `region`(地区) | ✅ | |
| `price`(价格) | ✅ | |
| `tags`(标签) | ✅ | |
| `photo_file_id`(图片) | ✅ | |
| `button_url`(链接) | ❌ | 链接由管理员控制防止恶意引流行为 |
| `button_text` | ✅ | |
| `is_active`(启用状态) | ❌ | 启停由管理员控制 |

#### 2.3.3 修改流程(立即生效模式)

```
老师私聊 bot → 点击"我的资料" → 看到当前所有字段
        ↓
点击"修改 [字段]" → bot 提示输入新值(同 v1 添加老师的 FSM)
        ↓
老师输入新值并确认
        ↓
bot 同时执行两个动作:
   ① UPDATE teachers SET [字段] = 新值
       → 立即在所有展示场景生效
   ② INSERT INTO teacher_edit_requests
       (teacher_id, field, old_value, new_value, status='pending')
        ↓
通知所有管理员"老师 X 修改了 [字段],原值: ..., 新值: ..."
```

> **"立即生效"的语义**:用户/搜索看到的**永远是 teachers 表当前值**,不存在"待审核版本"和"线上版本"两个并行视图。审核仅用于**事后回滚**。
>
> **例外:`photo_file_id`(图片)走"延后生效"——见 §2.3.3a**。

#### 2.3.3a 图片字段例外:延后生效模式

图片(`photo_file_id`)是**唯一**走延后生效的字段。原因:图片风控价值最大(违规风险高、视觉冲击直接),不应让未审核的内容直接对用户可见。

**修改流程**:

```
老师在自助菜单点击"修改图片"
        ↓
上传新图片 → bot 取 file_id
        ↓
bot 只执行 ① ②(不执行 ③):
   ①  ❌  不动 teachers.photo_file_id (保持旧图展示)
   ②  ✅  INSERT INTO teacher_edit_requests
       (field='photo_file_id', old_value=旧 file_id, new_value=新 file_id, status='pending')
   ③  ❌  无需 UPDATE teachers
        ↓
通知所有管理员;同时给老师反馈:
   "✅ 图片已提交审核,审核通过后立即生效;
    在此期间,展示位仍使用旧图"
```

**审核动作的字段分支**:

| 字段类型 | 通过(approve) | 驳回(reject) |
|---|---|---|
| 文字字段(5 个):display_name / region / price / tags / button_text | 不动 teachers(已是新值) | UPDATE teachers SET <field> = old_value |
| **图片字段**:photo_file_id | **UPDATE teachers SET photo_file_id = new_value** | 不动 teachers(本来就没改) |

**字段行为对照**:

| 阶段 | 5 个文字字段 | photo_file_id |
|---|---|---|
| 老师提交修改时 teachers 表 | 立即变成 new_value | 保持 old_value |
| pending 期间用户看到 | 新值 | 旧图 |
| 通过后 teachers 表 | 已是 new_value(不动) | 切换为 new_value |
| 驳回后 teachers 表 | 回滚到 old_value | 保持 old_value(已经是的旧值) |

**实现含义**(连锁改动):

- `approve_edit_request(request_id, reviewer_id)`:若 field_name == 'photo_file_id',额外执行 UPDATE teachers SET photo_file_id = new_value
- `reject_edit_request(request_id, reviewer_id, reason)`:若 field_name == 'photo_file_id',跳过 teachers 回滚(本来就没改)
- 老师"修改图片"FSM 在确认页明示:"图片将在审核通过后生效"

#### 2.3.4 审核界面

管理员后台增加"审核中心":

- 主面板加入口:`[📝 待审核 (N)]`(N = pending 状态的修改数)
- 点击进入后,按时间倒序展示每条修改:

```
📝 待审核修改

[1/3] 夏亦菲 修改了"价格"
原值: 1000P
新值: 1200P
修改时间: 2025-05-11 13:42

[✅ 通过] [❌ 驳回]

[⏭️ 下一条] [🔙 返回]
```

#### 2.3.5 审核动作语义

**通过(approve)**:
- `teacher_edit_requests.status` 改为 `approved`
- 不动 `teachers` 表(已经是新值)
- 不通知老师

**驳回(reject)**:
- `teacher_edit_requests.status` 改为 `rejected`
- `teachers.<field>` 回写为该条记录的 `old_value`
- 可选填驳回原因(`reject_reason`),非必填
- 通知老师"你的 [字段] 修改已被驳回(原因:XXX 或无原因)",私聊推送

#### 2.3.6 频次与限制

- **不限制修改频次**——老师一天可改任意次
- **每次修改独立成一条 `teacher_edit_requests` 记录**
- 如果同一字段在审核前被多次修改(例如:`1000P → 1100P → 1200P` 三次提交),驳回最新一条 `1200P`,会回滚到 `1100P`(它的 old_value),而非直接回到原始的 `1000P`
  - 这就是"回滚到上一版"的含义
  - 中间记录 `1000P → 1100P` 仍为 pending,管理员可继续审核
- **简化策略**:管理员对同一字段建议**按时间倒序逐条处理**(系统不强制,但 UI 上提示)

#### 2.3.7 边界情况

| 情况 | 处理 |
|---|---|
| 老师同一字段连续修改未审核 | 每次都创建新记录,审核员逐条处理 |
| 管理员通过修改后,老师又改了 | 已通过的记录归档,新记录进入 pending |
| 已驳回的字段又被老师改回相同值 | 视为新一次修改,创建新 pending 记录 |
| 老师被停用期间提交修改 | 停用 = `is_active=0`,但**不阻止**修改,审核通过/驳回逻辑不变 |
| 管理员驳回操作失败(数据库异常) | 记录失败日志,前端提示"操作失败,请重试",状态保持 pending |

### 2.4 F4:私聊组合搜索

#### 2.4.1 业务目标

让用户在**私聊场景**用更灵活的查询方式找到老师,弥补群组精准匹配的局限。**群组关键词响应行为不变**(精准匹配)。

#### 2.4.2 搜索方式:智能 AND

- **同类型 OR**:同一类型的多个值,只要命中一个即可
- **跨类型 AND**:不同类型的条件必须同时满足

**类型识别**:bot 自动判断每个输入词属于哪个类型(标签/地区/价格)。判断方式:

- 在 `teachers.tags` 中出现 → 标签
- 在 `teachers.region` 中出现 → 地区
- 在 `teachers.price` 中出现 → 价格
- 同一个词在多个类型中出现 → 视为命中所有类型(罕见,但要兜底)

**示例**:

| 输入 | 类型识别 | 等价 SQL 逻辑 |
|---|---|---|
| `御姐` | 标签 | `tags LIKE '%御姐%'` |
| `御姐 颜值` | 标签 OR 标签 | `tags LIKE '%御姐%' OR tags LIKE '%颜值%'`(同类型 OR) |
| `御姐 1000P` | 标签 + 价格 | `tags LIKE '%御姐%' AND price = '1000P'`(跨类型 AND) |
| `御姐 颜值 天府一街` | 标签 OR 标签 + 地区 | `(tags LIKE '%御姐%' OR tags LIKE '%颜值%') AND region = '天府一街'` |

#### 2.4.3 艺名走独立路径

**艺名(`display_name`)不参与组合搜索**:

- 用户输入完全等于某老师的 `display_name` → **直接返回该老师卡片**(模式 A 行为,同 v1 群组关键词)
- 不论后续输入什么其他词,艺名命中优先

> **理由**:艺名唯一对应一位老师,加任何条件都只能筛同一位,组合搜索无意义。独立路径让"我就想看夏亦菲"的用户体验最直接。

**判定优先级**:

```
用户输入 → 完全匹配某 display_name?
              ├─ 是 → 返回该老师卡片(模式 A)
              └─ 否 → 进入组合搜索(模式 B)
```

#### 2.4.4 命中展示

| 命中情况 | 展示形式 |
|---|---|
| 0 老师 | "未找到匹配的老师,试试别的关键词?" |
| 1 老师 | 老师卡片(同 v1 模式 A) |
| 2-10 老师 | 超链接列表(同 v1 模式 B) |
| 10+ 老师 | 超链接列表 + "结果较多,试试加更多关键词?"提示;按签到时间或随机排序(实现时定) |

> **零结果回复 vs 群组的静默不回复**:私聊里用户主动发起搜索,零结果不回复会让人困惑;群里静默是为了防刷屏。两个场景规则不同,**私聊一定要回复**。

#### 2.4.5 群组关键词的关系

群组关键词响应**完全保留 v1 行为**:

- 仅响应配置的群组
- 仅精准匹配
- 不区分大小写
- 无匹配时静默
- 受冷却时间约束

**唯一变化**:模式 A 卡片(单老师)增加"⭐ 收藏"按钮(F1 入口)。模式 B 列表不变。

### 2.5 C1:`/start` 给非管理员的私聊冷启动改造

#### 2.5.1 v1 现状

`/start` 仅对管理员有响应,普通用户发 `/start` 可能无回应或返回管理员菜单(看现有实现)。

#### 2.5.2 v2 改造目标

让普通用户私聊 `/start` 时看到一个"消费入口"主面板。

#### 2.5.3 主菜单结构

```
👋 欢迎使用痴颜录 Bot

[📚 今日开课老师]
[⭐ 我的收藏]
[💝 收藏开课]
[🔍 搜索老师]
```

四个按钮的含义:

| 按钮 | 内容 |
|---|---|
| 📚 今日开课老师 | 当天所有已签到老师(同 14:00 频道发布的内容) |
| ⭐ 我的收藏 | 当前用户的收藏老师列表(可点击查看每位 / 取消收藏) |
| 💝 收藏开课 | 收藏 ∩ 已签到 老师(F2 通知里的内容,但用户主动查询时也可用) |
| 🔍 搜索老师 | 进入搜索引导:"请输入关键词,例如 御姐 1000P" → 等待用户输入 |

#### 2.5.4 子菜单

每个子菜单按钮内进入对应展示,所有子菜单都有"🔙 返回主菜单"按钮。

**📚 今日开课老师**:
- 0 位 → "今天暂无老师开课"
- 1 位+ → 按钮列表(同 14:00 频道格式),每位老师一个按钮,跳转 button_url

**⭐ 我的收藏**:
- 0 位 → "你还没有收藏任何老师,试试 🔍 搜索老师 找一个"
- 1 位+ → 老师列表(每行一个老师 + 取消收藏按钮)
- 点击老师可展开完整卡片(单独发一条消息)

**💝 收藏开课**:
- 0 位 → "今天你的收藏老师没有人开课"
- 1 位+ → 同 F2 推送内容(按钮列表)

**🔍 搜索老师**:
- 进入 FSM 状态等待输入
- 用户发文字 → 调用 F4 搜索流程
- 用户发 `/cancel` 或点击"返回" → 退出搜索

#### 2.5.5 与管理员入口的关系

- 如果 `/start` 用户是**管理员**(超管或普通管理员):**展示管理面板**(v1 行为不变)
- 如果是**老师**(在 `teachers` 表中,**含已停用 `is_active=0` 的老师**):**展示老师私聊菜单**
- 否则:展示本节定义的非管理员菜单

**老师私聊菜单内容**:

```
👤 你好,{display_name}

[✏️ 我的资料]    ← F3 自助管理入口
[✅ 今日签到]    ← 等价于发文字"签到",二者并存
[🔙 退出]
```

- **停用老师仍可进入菜单**:可继续修改资料(等待管理员审核或恢复启用),签到操作仍允许(v1 行为)
- **签到入口并存**:菜单的 `[✅ 今日签到]` 按钮和 v1 的"发送'签到'文字"两种触发方式同时生效,行为完全一致;**不强制老师只用其中一种**

> **多角色重叠**:如果一个用户同时是管理员和老师,优先展示管理员菜单(权限高优先)。或者在管理员菜单里加一个"切换到老师视图"按钮——**实现时定**。

## 3. 数据库 Schema 增量

### 3.1 新增表

#### 3.1.1 `users` 表(普通用户)

```sql
CREATE TABLE users (
    user_id INTEGER PRIMARY KEY,           -- Telegram user_id
    username TEXT,                          -- Telegram username(可空)
    first_name TEXT,                        -- 用于通知 mention 的展示名
    last_started_bot INTEGER DEFAULT 0,     -- 0=从未私聊;1=已私聊过(可推送)
    notify_enabled INTEGER DEFAULT 1,       -- 用户级通知开关(预留)
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    last_active_at TEXT                     -- 最后活跃时间(用于识别死号)
);
```

**何时插入**:
- 用户首次私聊 bot(发送任何消息或 `/start`)→ INSERT/UPDATE
- 用户在群里点收藏按钮(若未私聊过) → INSERT 但 `last_started_bot=0`
- 用户从 deep link 进入 bot → UPDATE `last_started_bot=1`

#### 3.1.2 `favorites` 表(收藏关系)

```sql
CREATE TABLE favorites (
    user_id INTEGER NOT NULL,               -- 收藏者
    teacher_id INTEGER NOT NULL,            -- 被收藏老师(teachers.user_id)
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (user_id, teacher_id),
    FOREIGN KEY (teacher_id) REFERENCES teachers(user_id) ON DELETE CASCADE
);

CREATE INDEX idx_favorites_teacher ON favorites(teacher_id);
-- 用于 F2:按老师反查"谁收藏了我"
CREATE INDEX idx_favorites_user ON favorites(user_id);
-- 用于 C1 "我的收藏"列表
```

**约束**:
- 复合主键保证幂等(同用户对同老师只能收藏一次)
- 老师删除时级联清理收藏关系(防止遗留无效引用)

#### 3.1.3 `teacher_edit_requests` 表(修改记录)

```sql
CREATE TABLE teacher_edit_requests (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    teacher_id INTEGER NOT NULL,
    field_name TEXT NOT NULL,               -- 改的字段(display_name/region/price/tags/photo_file_id/button_url/button_text)
    old_value TEXT,                          -- 改前的值(用于回滚,nullable 因为图片可为空)
    new_value TEXT,                          -- 改后的值
    status TEXT NOT NULL DEFAULT 'pending', -- pending / approved / rejected
    reviewer_id INTEGER,                     -- 审核管理员 ID(approve/reject 时填入)
    reject_reason TEXT,                      -- 驳回原因(可选)
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    reviewed_at TEXT,                        -- 审核时间
    FOREIGN KEY (teacher_id) REFERENCES teachers(user_id)
);

CREATE INDEX idx_edit_requests_status ON teacher_edit_requests(status);
-- 用于审核界面查 pending 列表
CREATE INDEX idx_edit_requests_teacher ON teacher_edit_requests(teacher_id, created_at);
-- 用于查某老师的修改历史
```

**字段语义**:
- `old_value`:**修改时刻**老师该字段的当前值(后续多次修改时,每条记录的 old_value 是它自己时刻的当前值,不是基线值)
- `new_value`:用户提交的新值
- 表中数据保留(归档),不删除

**审核动作对 teachers 表的影响**(v2 §2.3.3 + §2.3.3a):

| field_name | approve 时 | reject 时 |
|---|---|---|
| 文字字段(display_name / region / price / tags / button_text) | 不动(teachers 提交时已立即更新为 new_value) | `UPDATE teachers SET <field> = old_value WHERE user_id = teacher_id` |
| **photo_file_id**(图片例外) | `UPDATE teachers SET photo_file_id = new_value WHERE user_id = teacher_id` | 不动(teachers 提交时本来就没更新,保持旧图) |

### 3.2 现有表的修改

无现有表的破坏性变更。仅可能新增索引(实现时按性能需求评估)。

### 3.3 数据迁移

无历史数据迁移需求——v2 表都是新增,v1 老用户/老师/签到/管理员数据原封不动。

唯一例外:`users` 表是新加的,但已有的"管理员"和"老师"在另两张表里。**普通用户数据从 v2 上线开始累积**,老用户首次私聊 bot 时插入 `users` 记录。

### 3.4 隐私与合规

- `users.first_name` 来自 Telegram,不需用户授权,但作为 PII 应注意备份/导出场景的脱敏
- `favorites` 表暴露用户偏好,**老师本人不可查询**(仅管理员后台可见聚合数据,不暴露具体收藏者)
- 管理员后台导出数据时,涉及 `users` 表需加二次确认或限制导出范围

## 5. 实施顺序建议

### 5.1 模块依赖图

       [数据库 schema 增量]

   ┌──────────┴──────────┐

### 5.2 推荐五步推进

| 步骤 | 内容 | 大致工作量 |
|---|---|---|
| 1 | 数据库 schema 增量(`users` / `favorites` / `teacher_edit_requests`)+ 对应 CRUD 函数 | 小 |
| 2 | C1 `/start` 非管理员菜单 + F4 私聊搜索(智能 AND + 艺名独立路径) | 中 |

### 5.3 上线前的灰度建议

## 6. 决策记录

> 关键产品决策与理由,实现时如有疑问回查此处而非临时拍板。

| # | 决策 | 理由 |
|---|---|---|
| 1 | 通知时机选 14:00 统一推送(非签到瞬间触发) | 集中发送,用户体感统一;签到瞬间触发会形成"零散推送",打扰感强 |
| 2 | 通知聚合(每用户一条) | 减少推送条数 + 用户体验佳;真正的负荷优化在"按用户聚合",非"模板复用" |
| 3 | 全员未签到时静默 | 防打扰;让 bot 在闲置时不刷存在感 |
| 4 | 收藏按钮**不加**到频道发布 | 频道作为"广播导流"形态,按钮密度低更易扫读;收藏是主动意图,放在搜索场景更合理 |
| 5 | 群组收藏使用混合 deep link 流 | 既保证收藏意图不流失(立即存),又引导未私聊用户激活通知能力 |
| 6 | 收藏不开放命令入口 | 命令认知成本高;按钮直观 |
| 7 | 已收藏卡片显示 `✅ 已收藏(点击取消)`(**仅私聊场景**) | 状态可见 + 取消便捷;不需要二次确认(操作可恢复) |
| 7b | 群组场景按钮**恒为 `⭐ 收藏`**,操作结果通过 alert 反馈 | 群组卡片对所有人可见,无法按个体渲染状态;alert 是私有反馈通道,既能告知当前用户结果又不污染群组消息 |
| 7c | `button_url` 老师**不可自助修改**,仅管理员可改 | 防止恶意引流(改成钓鱼/广告/违规链接);其余 6 个字段可自助 |
| 8 | 老师自助修改"立即生效 + 事后审核" | 体验优先于风控(本平台修改频次低,风险可控) |
| 9 | 不限制修改频次 | 信任老师;频次限制带来的复杂度大于收益 |
| 10 | 驳回回滚到"上一版本"(不是基线) | 简化语义;多次修改时按时间倒序逐条处理 |
| 11 | 驳回原因可选 | 减少管理员摩擦;有原因更友好,但不强制 |
| 12 | 搜索方式选"智能 AND" | 最贴近用户直觉(同类型 OR、跨类型 AND);AND/OR 二元方案要么过于精准要么过于宽松 |
| 13 | 艺名走独立路径,不参与组合搜索 | 艺名唯一对应一位老师,加任何条件都只能筛同一位;独立路径让"我就想看 X"的体验最直接 |
| 14 | `/start` 非管理员展示三块内容 | 覆盖三种最高频场景:看今天谁开课、看自己关注的、查具体老师 |
| 15 | 停用老师仍可进入自助菜单,继续改资料 | 停用可能是"暂时下线",非永久状态;允许修改让老师能在停用期间维护资料,等管理员恢复启用 |
| 16 | 签到入口"按钮"和"发文字"并存 | 兼顾新老用户习惯,不强制行为变更;按钮降低误操作率,文字保留兼容 |
| 17 | **图片字段例外**:photo_file_id 单独走"延后生效",其他 5 个文字字段仍立即生效 | 图片是违规风险最高的字段(视觉冲击直接、可能不雅);不让未审核内容对用户可见。其他字段错改影响小,立即生效保留体验。带来的代价:approve/reject 函数对 photo_file_id 特殊分支处理 |
