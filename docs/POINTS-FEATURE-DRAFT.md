## 0. 一句话目标

为用户成功提交并通过审核的报告自动加积分，作为社区活跃度的可视化激励。积分由审核员根据**约课截图 + 现场手势 + 过程描述**等审核材料推断套餐类型后手动赋予。

> ⚠️ **加分对象明确**：积分加给**报告提交者（user）**，不加给被评价的老师（teacher）。老师的"等级"由其报告聚合的综合评分 + 三级评级分布反映（详见 REVIEW spec），不另设老师积分。

## 1. 核心规则

### 1.1 加分时机

**仅在「报告审核通过」时由超管手动选择加多少分。**

审核流程衔接（覆盖 REVIEW 文档 §4.4「通过流程」）：

```
原来：
  [✅ 通过] → 立即 DB approved + 频道发评 + 通知评价者

现在：
  [✅ 通过] → 进入加分子页 → 超管选加分值 → 一次性提交：
    1. DB approved + reviewer_id + reviewed_at
    2. point_transactions 加一条记录
    3. users.total_points += delta
    4. 频道发评 + 档案帖统计更新
    5. 通知评价者（含本次积分增量 + 当前总积分）
```

### 1.2 套餐对应建议分值（审核员参考）

| 套餐类型 | 建议加分 | 说明 |
|---|---|---|
| P / PP | **+1** | 默认（最常见） |
| 包时（按小时） | **+3** | |
| 包夜 | **+5** | |
| 包天 | **+8** | |
| 不加分 | **+0** | 用于审核通过但材料质量不足以判断套餐时 |
| 自定义 | 任意非负整数 | 极少使用，用于特殊情况 |

> 这些值仅作为按钮预设，**审核员根据用户的过程描述 + 上传截图自行判断**实际套餐类型。

### 1.3 扣分

**本期不实现扣分功能**。所有积分单向递增。

未来可能的扣分场景（不在本期）：
- 用户违规被处罚
- 抽奖消耗（若后续开放积分门票制抽奖）
- 积分商城兑换

### 1.4 积分用途（本期仅作为荣誉激励）

- 用户私聊主菜单新增 [💰 我的积分] 入口（**仅在此页面展示积分**）
- **不在老师详情页 / 评论区 / 任何公共页面展示用户的积分** —— 保持隐私
- **本期暂不与抽奖耦合**，抽奖参与门槛是关注频道（详见 LOTTERY-FEATURE-DRAFT.md）
- **本期暂不做积分排行榜**

未来可扩展用途（不在本期）：
- 积分排行榜
- 积分商城（兑换实物 / 老师推荐位等）
- 高积分用户特权（如优先客服）
- 抽奖消耗

## 3. 管理员侧

### 3.1 审核通过子页（关键流程变化）

在 REVIEW 文档 §4.2 审核详情页基础上，**修改 [✅ 通过] 按钮的行为**：

点 [✅ 通过] 后**不立即提交**，而是进入加分选择子页：

```
[积分选择子页 - 审核通过]

老师：丁小夏
评价者：小* (uid: ****6204)
当前用户总积分：80

请根据审核材料给该用户加分：

[+1 P / PP]    [+3 包时]
[+5 包夜]      [+8 包天]
[+0 不加分]    [💬 自定义]

[⬅️ 取消通过，返回审核]
```

选完任一加分按钮 → 一次性 commit 所有操作（详见 §1.1）+ 返回审核列表下一条。

选 [💬 自定义] → FSM 等待用户输入整数（0-100）。

### 3.2 手动加分 / 扣分（管理员后台工具）

主面板新增 [💰 积分管理]（仅超管可见）：

```
💰 积分管理

[🔍 查询用户积分]
[➕ 手动加分]
[📊 积分总览]      ← 显示总用户数 / 总加分 / TOP 10 用户
[🔙 返回主面板]
```

#### 查询用户积分
输入 user_id 或 @username → 显示该用户的总积分 + 最近 10 条明细。

#### 手动加分（全按钮式，无文字输入）

FSM 流程：

```
[Step 1] 输入要加分的用户 ID 或 @username（文字输入）
[Step 2] 选择加分值（按钮）：
  [+1 P / PP]    [+3 包时]
  [+5 包夜]      [+8 包天]
  [+10]          [+20]
  [➖ 扣分]      [💬 自定义]
  [❌ 取消]

  扣分 → 进 Step 2b：[-1] [-3] [-5] [-10] [💬 自定义负数]
  自定义 → 进 Step 2c：文字输入整数（-100 到 100）

[Step 3] 选择加分原因（按钮，不再用文字输入）：
  [📝 报告审核补加]
  [🎁 活动奖励]
  [⚠️ 违规扣分]
  [🛠 系统修正]
  [💬 自定义]      ← 文字输入
  [❌ 取消]

[Step 4] 确认页：
  老师将给 用户 X (****6204) 加 +5 分
  原因：报告审核补加
  当前余额：80 → 85

  [✅ 确认] [❌ 取消]
```

仅超管可用。所有手动加分都写 admin_audit_logs（含 operator_id / delta / reason）。

## 4. 数据模型

### 4.1 `users` 表扩展

```sql
ALTER TABLE users ADD COLUMN total_points INTEGER DEFAULT 0;
```

`total_points` = 该用户所有 point_transactions 的 delta 累计（可由触发器或应用层维护；本期应用层维护更简单）。

### 4.2 新表 `point_transactions`

```sql
CREATE TABLE point_transactions (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id      INTEGER NOT NULL,
    delta        INTEGER NOT NULL,                   -- 正数加分，负数扣分
    reason       TEXT NOT NULL,                      -- 'review_approved' / 'admin_grant' / ...
    related_id   INTEGER,                            -- 关联的 review_id 或其他业务 id
    operator_id  INTEGER,                            -- 操作管理员 ID（NULL 表示系统）
    note         TEXT,                               -- 备注（管理员手动加分时填）
    created_at   TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_point_tx_user_time ON point_transactions(user_id, created_at);
CREATE INDEX idx_point_tx_related ON point_transactions(reason, related_id);
```

### 4.3 配置常量

```python
# bot/database.py
POINT_PACKAGE_OPTIONS = [
    {"key": "p",      "label": "P / PP",  "delta": 1},
    {"key": "hour",   "label": "包时",    "delta": 3},
    {"key": "night",  "label": "包夜",    "delta": 5},
    {"key": "day",    "label": "包天",    "delta": 8},
    {"key": "zero",   "label": "不加分",  "delta": 0},
]
POINT_CUSTOM_MIN = 0
POINT_CUSTOM_MAX = 100
```

## 6. 边界情况

| 场景 | 处理 |
|---|---|
| 审核员选 [+0 不加分] | DB 仍写一条 delta=0 的 transaction（便于追溯）|
| 用户被删除 | point_transactions 保留（只读历史）|
| 报告 status 从 approved 改回 rejected（极端管理员操作）| **本期不允许**，approved 是终态 |
| 用户积分查询时 transactions 表特别大 | 用 index + LIMIT 20 分页 |
| 手动加分输入负数 | 视为扣分；扣分使用户总积分变负 → 拒绝 + 提示 |
| 自定义加分输入非整数 / 超 100 | 校验失败，停留 FSM |
